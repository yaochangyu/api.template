## P0-2：PipeWriter.UnflushedBytes 回歸（獨立於 P0-1）
【事實】GET `/api/v1/tests`（`TestController.cs:8-19`，純 `Ok(匿名物件)`，無 DI 疑慮）整合測試 500；`logs/aspnet-202607120112.txt:46-48,190-192`：
`The PipeWriter 'ResponseBodyPipeWriter' does not implement PipeWriter.UnflushedBytes.`（STJ SerializeAsync → SystemTextJsonOutputFormatter）
【根因】（主對話確診，見 findings-build-test.md）`IntegrationTest.csproj:17` Mvc.Testing **8.0.10** 的 TestServer 未實作 net10.0 STJ 要求的 `PipeWriter.UnflushedBytes`。僅測試環境受影響，Kestrel runtime 無虞。
【交叉解釋】P0-1 的 member 端點：原始例外是 DI，但 ExceptionHandlingMiddleware 之後 MVC 寫回 JSON 時再觸發 PipeWriter → 測試端只看到 PipeWriter 堆疊，**遮蔽了 DI 真因**——這正是 .NET 10 升級時誤判成序列化問題、繞遠路的機制。
【嚴重度】P0

## P1：API First / Code First 混用（P0-1 的架構性根因）
【事實】`doc/openapi.yml` 仍定義 v2 paths → NSwag 全量生成含 v2 的 API First Controller；csproj 未排除 `Contract/AutoGenerated/*.cs` 編譯 → 同一組件內 v2 資源存在兩套 Controller（手寫 Code First＋生成 API First）。CLAUDE.md 明文「禁止混用」。
【推論】光改 DI 不能根治；需二選一：v2 從 openapi.yml 移除（生成範圍只剩 v1），或 v2 走滿 API First。
【嚴重度】P1
