# 開發規則

此檔案包含 API 範本專案的核心開發規則、架構指南與強制詢問情境。  
適合：所有開發者、SKILL 使用者、測試工程師。

---

## 開發指令

### Taskfile 使用原則
- **優先使用 Taskfile**: 所有重複執行的開發指令應盡可能透過 `task` 命令執行
- **命令集中管理**: 複雜的多步驟指令應寫入 `Taskfile.yml`

### 常用指令
- **開發模式執行 API**: `task api-dev`
- **建置解決方案**: `task build`
- **執行單元測試**: `task test-unit`
- **執行整合測試**: `task test-integration`
- **產生 API 程式碼**: `task codegen-api`
- **從資料庫反向工程產生實體**: `task ef-codegen`
- **建立新的 Migration**: `task ef-migration-add MIGRATION_NAME=<MigrationName>`
- **更新資料庫**: `task ef-database-update`

**重要**: EF Core 相關指令必須透過 Taskfile 執行。

## 架構概述

### 核心專案
- **JobBank1111.Job.WebAPI**: 主要的 Web API 應用程式
- **JobBank1111.Infrastructure**: 跨領域基礎設施服務
- **JobBank1111.Job.DB**: Entity Framework Core 資料存取層
- **JobBank1111.Job.Contract**: 從 OpenAPI 規格自動產生的 API 客戶端合約

### 分層架構

**Controller → Handler → Repository** 模式：
- **Controller 層**: HTTP 請求/回應、路由、請求驗證、HTTP 狀態碼對應
- **Handler 層**: 核心業務邏輯、流程協調、錯誤處理與結果封裝
- **Repository 層**: 資料存取邏輯、EF Core 操作、資料庫查詢封裝

### 組織方式
- **方案 A：單一專案結構** — 所有功能層在 `JobBank1111.Job.WebAPI` 內（適合小型團隊）
- **方案 B：多專案結構** — Controller、Handler、Repository 各自獨立專案（適合大型團隊）

### 技術堆疊
- **框架**: ASP.NET Core 10
- **資料庫**: Entity Framework Core 與 SQL Server
- **快取**: Redis 搭配記憶體內快取備援
- **錯誤處理**: CSharpFunctionalExtensions (Result Pattern)
- **驗證**: FluentValidation
- **日誌記錄**: Serilog 結構化日誌
- **測試**: xUnit、Testcontainers、Reqnroll (BDD)
- **API 文件**: Swagger/OpenAPI 搭配 ReDoc、Scalar

### 程式碼產生工作流程
1. API 規格維護在 `doc/openapi.yml`
2. 使用 Refitter 產生客戶端程式碼至 `JobBank1111.Job.Contract`
3. 使用 NSwag 產生伺服器控制器至 `JobBank1111.Job.WebAPI/Contract`
4. 使用 EF Core 反向工程產生資料庫實體至 `JobBank1111.Job.DB`

**重要規範**: 
- 所有自動產生的程式碼都放在 `AutoGenerated` 資料夾中，不可手動編輯
- EF Core 反向工程與 Migration 必須透過 Taskfile 執行

## 強制詢問情境（必須詢問，不能跳過）

### 1️⃣ 專案初始化與配置
- 資料庫類型選擇（SQL Server / PostgreSQL / MySQL）
- Redis 快取需求（yes / no）
- 專案結構組織（single-project / multi-project）
- 是否使用 GitHub 範本

### 2️⃣ 資料庫相關操作
- Code First vs Database First 模式
- Migration 名稱與套用策略
- 資料表範圍選擇

### 3️⃣ 功能實作時必須詢問
**a) API 開發流程**
- ✅ API First（推薦）：定義 OpenAPI → 產生骨架 → 實作 → 文件同步
- ✅ Code First：直接實作 → 手動維護 OpenAPI
- ⚠️ 同一專案內不得混用兩種流程。**範本例外**：本範本刻意以 v1 示範 API First、v2 示範 Code First 供教學對照；由此範本建立的實際專案仍須全專案擇一

詳細決策邏輯見 [決策框架 - API 開發流程](./decision-framework.md#api-開發流程決策)

**b) OpenAPI 規格狀態**（僅 API First 時）
- 已定義 | 需要更新 | 尚未定義

詳細決策邏輯見 [決策框架 - OpenAPI 狀態評估](./decision-framework.md#openapi-規格狀態評估)

**c) 需要實作的分層**
- Controller | Handler | Repository（可多選）

### 4️⃣ **測試策略詢問**（最重要，不能假設）
**問 a)：是否需要測試？**
- ✅ 完整測試（BDD + 單元測試）
- ✅ 僅 BDD 整合測試
- ✅ 僅單元測試
- ❌ 暫不實作

詳細決策邏輯見 [決策框架 - 測試策略決策樹](./decision-framework.md#測試策略決策樹)

**問 b)：測試範圍**
- 完整功能 | 核心業務邏輯 | 關鍵路徑 | 異常情境

**問 c)：BDD 情境**（若選擇 BDD）
- 已有 .feature 檔案？
- 需要新增哪些情境？
- 需要 AI 協助 Gherkin 語法？

**問 d)：測試環境**
- Docker 容器（SQL Server / Redis）
- Seed Data | 動態產生 | 清理策略

**問 e)：測試方法**
- ✅ API 端點必須 BDD 測試（.feature + Reqnroll）
- ✅ 優先 Testcontainers（Docker）作為測試替身
- ✅ 禁止直接測試 Controller

### 5️⃣ 效能最佳化
- 優化面向選擇（資料庫查詢 / 快取 / 非同步 / 記憶體）

詳細決策邏輯見 [決策框架 - 效能優化決策](./decision-framework.md#效能優化決策)

## 禁止的行為 ❌

1. ❌ 擅自使用預設值 — 必須明確詢問
2. ❌ 跳過詢問步驟 — 即使有推薦選項，仍須確認
3. ❌ 一次詢問過多問題 — 最多 3-4 個
4. ❌ 提供不明確的選項 — 必須加說明
5. ❌ 假設測試需求 — 不可假設，必須明確詢問
6. ❌ 跳過測試實作詢問 — 實作新功能時必須詢問測試策略

## 最佳實踐檢查清單

### 開發前
- [ ] 檢查專案狀態（是否需要初始化）
- [ ] **【必須】詢問 API 開發流程選擇**（API First vs Code First）
- [ ] **【必須】詢問測試策略與範圍**（不可假設）
- [ ] 架構決策確認

### 開發中
- [ ] 所有 I/O 操作使用 async/await
- [ ] 傳遞 CancellationToken
- [ ] 使用 Result Pattern 處理錯誤
- [ ] 使用 DbContextFactory 而非直接注入 DbContext
- [ ] 主建構函式注入（Primary Constructor）
- [ ] 使用結構化日誌（Serilog）

### 測試（依使用者選擇執行）
- [ ] **確認已詢問使用者測試需求**
- [ ] 透過 BDD 情境測試 API（若需要 BDD）
- [ ] 實作單元測試（若需要單元測試）
- [ ] 使用 Docker 測試環境（Testcontainers）
- [ ] 避免直接測試 Controller
- [ ] 測試涵蓋核心業務邏輯與異常情境

### 部署前
- [ ] 檢查機密未存放在 appsettings.json
- [ ] 所有測試通過（若有實作測試）
- [ ] 無編譯警告
- [ ] 文件已更新
- [ ] Code Review 完成
