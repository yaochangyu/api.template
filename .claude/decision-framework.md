# 決策框架

此檔案提供關鍵工作流的決策邏輯與檢查清單。  
適合：特定工作流（API 開發、測試規劃、資料庫選擇、快取設計）。

---

## API 開發流程決策

### 流程選擇
兩種主流方法，各有優缺點：

| 比較項 | API First | Code First |
|--------|----------|-----------|
| **定義順序** | OpenAPI → 骨架 → 實作 | 實作 → 文件更新 |
| **文件同步** | 自動同步 | 手動維護 |
| **改動成本** | 需修改 OpenAPI + 代碼 | 只改代碼 |
| **適用情景** | 公開 API、多客戶端、明確規格 | 內部 API、快速迭代 |
| **推薦度** | ✅ 推薦（優先選擇） | ⚠️ 須詢問確認 |

### 決策樹
```
功能實作需求
  ├─ 是否有明確的 API 規格需求？
  │   ├─ 是 → 詢問「是否偏好 API First？」
  │   │   ├─ 是 → API First 流程
  │   │   └─ 否 → Code First 流程
  │   └─ 否 → 詢問「客戶端是否需要 OpenAPI 合約？」
  │       ├─ 是 → 使用 API First（先定義規格）
  │       └─ 否 → Code First（直接實作）
```

### API First 工作流
1. **定義 OpenAPI** (`doc/openapi.yml`)
   - 定義端點、請求/回應結構、狀態碼
2. **產生代碼骨架**
   - Controller: `task codegen-api`
   - Handler 簽名與 DTO
3. **實作業務邏輯** (Handler / Repository)
4. **文件保持同步**
   - API 變更 → 更新 OpenAPI
   - OpenAPI 變更 → 重新產生骨架

### Code First 工作流
1. **實作功能** (Controller → Handler → Repository)
2. **實作測試** (BDD + 單元測試)
3. **手動更新 OpenAPI**
   - 產生 NSwag 文件
   - 人工審查和調整
4. **文件維護責任**
   - API 改動 → 手動更新 OpenAPI

## OpenAPI 規格狀態評估

### 規格已定義
- OpenAPI 檔案存在且最近更新
- 所有端點均有明確文件
- 決策：直接使用現有規格

### 規格需要更新
- OpenAPI 檔案存在但過時（超過 1 sprint）
- 新功能未在規格中
- 決策：先更新 OpenAPI，再進行開發

### 規格尚未定義
- 新的 API 域名或完全新功能
- 決策：根據複雜度選擇：
  - **簡單 API**（< 5 端點）→ Code First 可接受
  - **複雜 API**（> 5 端點）→ 強烈建議 API First

## 測試策略決策樹

### 第 1 層：需要測試嗎？

```
新功能實作
  ├─ 是否觸及業務邏輯？
  │   ├─ 是 → 進入第 2 層決策
  │   └─ 否 → 考慮跳過（但需確認是否純粹資料傳遞）
```

### 第 2 層：測試類型組合

| 選擇 | 包含內容 | 適用場景 | 工時 |
|------|---------|---------|------|
| **完整測試** | BDD + 單元測試 | 關鍵業務邏輯、公開 API | 中 |
| **僅 BDD** | 整合測試（API 端點） | 快速驗證、端對端需求 | 低 |
| **僅單元測試** | Handler / Repository 測試 | 複雜業務邏輯、內部服務 | 中 |
| **無測試** | 文件 + 手動驗證 | 臨時代碼、原型開發 | 低 |

### 第 3 層：測試範圍

**完整功能**
- 所有正常路徑
- 所有異常路徑
- 邊界值

**核心業務邏輯**
- 主流程
- 關鍵異常（如驗證失敗、資源不存在）

**關鍵路徑**
- Happy Path
- 最常見的異常情況

**異常情境**
- 僅錯誤處理、邊界值、邏輯分支

## 測試實作詳細決策

### BDD 測試（`.feature` 檔案）

**何時編寫**
- [ ] API 端點需要整合測試
- [ ] 需要業務規則驗證
- [ ] 需要文件化使用者故事

**環境選擇**
- [ ] Docker Testcontainers（優先）
  - 真實 SQL Server / Redis
  - 資料隔離
  - 環境一致性
- [ ] Seed Data 策略
  - 靜態資料：用於通用測試
  - 動態產生：用於數據變異測試

**禁止項目**
- ❌ 直接測試 Controller 類
- ❌ 使用 Mock 資料庫（除非無法使用 Docker）
- ❌ 跳過清理步驟（測試間資料污染）

### 單元測試

**何時編寫**
- [ ] Handler 中包含複雜業務邏輯
- [ ] Repository 中有複雜的查詢
- [ ] 涉及外部服務呼叫

**測試對象**
- ✅ Handler（業務邏輯、流程協調）
- ✅ Repository（查詢邏輯、資料轉換）
- ❌ Controller（改用 BDD 測試）

**替身策略**
- [ ] DbContext → Mock（無法使用 Docker 時）
- [ ] 外部服務 → Mock（如 HTTP 客戶端）
- [ ] 資料庫查詢 → 優先用真實 DB（Testcontainers）

## 資料庫選擇決策

### 初始化時
**詢問問題**
- [ ] 現有項目是否有指定資料庫？
- [ ] 是否需要多資料庫支援？
- [ ] 效能要求如何？

**選擇標準**

| 資料庫 | 適用場景 | EF Core 支援 |
|--------|---------|------------|
| **SQL Server** | 企業應用、財務系統 | ✅ 完全支援 |
| **PostgreSQL** | 開源、高併發 | ✅ 完全支援 |
| **MySQL** | 成本考量、共享主機 | ✅ 完全支援 |

### 模式選擇
- **Code First** — 推薦（版本控制友善）
- **Database First** — 現有資料庫用此

### EF Core 遷移管理
```
建立 Migration
  ├─ task ef-migration-add MIGRATION_NAME=AddUserTable
  └─ 檢查自動產生的程式碼
         ├─ 業務邏輯？ → 手動調整
         └─ 資料型別？ → 驗證精確性

套用 Migration
  ├─ 開發環境：task ef-database-update
  └─ 生產環境：手動檢查，確認備份
```

## 快取策略決策

### 快取層次架構

```
快取需求評估
  ├─ 單機還是分散式？
  │   ├─ 單機 → L1 記憶體快取（IMemoryCache）
  │   └─ 分散式 → L1 + L2 (Redis)
  │
  └─ 資料特性是什麼？
      ├─ 實時資料 → 無快取或極短 TTL
      ├─ 查詢結果 → L1 記憶體快取
      ├─ 跨服務資料 → L2 Redis
      └─ 靜態資料 → 長期快取（小時/天）
```

### 快取失效策略

| 策略 | 觸發條件 | 場景 |
|------|---------|------|
| **TTL（時間失效）** | 設定過期時間 | 實時性低的資料 |
| **主動失效** | 資料變更時清除 | 讀多寫少 |
| **事件失效** | 監聽領域事件 | 複雜業務邏輯 |
| **無效失效** | 信任過期時間 | 非關鍵資料 |

### 實作檢查清單
- [ ] 定義快取的 Key 命名規範
- [ ] 設定合理的 TTL
- [ ] 實作快取失效機制
- [ ] 監控快取命中率
- [ ] 測試快取失效邊界情況

## 效能優化決策

### 優化優先級

```
效能問題排查
  1. 分析問題根源
     ├─ 資料庫查詢（N+1、缺乏索引）
     ├─ 快取缺失（冷啟動、失效過頻繁）
     ├─ 同步等待（未使用 async）
     └─ 記憶體洩漏
  
  2. 確定優化面向（優先順序）
     ├─ 資料庫查詢優化 P0
     ├─ 快取設計優化 P1
     ├─ 非同步優化 P2
     └─ 記憶體優化 P3
```

### 資料庫查詢優化
- [ ] 檢查 N+1 問題（用 .Include()）
- [ ] 驗證索引存在
- [ ] 選擇性查詢（Select 需要的欄位）
- [ ] 分頁（大量資料集）
- [ ] 查詢快取（EF Core 計畫快取）

### 非同步優化
- [ ] 所有 I/O 操作：async/await
- [ ] 傳遞 CancellationToken
- [ ] 避免 .Result / .Wait()

### 快取優化
見 [快取策略決策](#快取策略決策) 章節
